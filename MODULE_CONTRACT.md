# MODULE_CONTRACT.md — контракт модуля

> Как модуль регистрируется в ядре и общается с ним. Заполняется на шаге 1.2 (🔴 Claude).
> Статус: ЗАГОТОВКА — принципы зафиксированы, точный интерфейс детализируется при сборке ядра.

## Зачем
Ядро не знает о конкретных модулях. Каждый модуль реализует единый контракт и регистрируется в реестре. Это позволяет добавлять/дорабатывать модули, не трогая ядро и соседей.

## Что определяет контракт (намечено)
- **commands** — какие команды обслуживает модуль (`/task`, `/note`, `/kb`…).
- **canHandle(intent)** — может ли модуль обработать распознанное намерение.
- **handle(ctx)** — выполнить действие. `ctx` даёт доступ к: пользователю, воркспейсу, сессии, payload, языку (для i18n), точке проверки доступа (gate).

## Сквозные требования к каждому модулю
1. **Логика не в боте.** Модуль вызывает общий слой логики, не дублирует её. Бот/сайт/приложение должны мочь переиспользовать ту же логику.
2. **Воркспейс и изоляция.** Любой доступ к данным — в рамках `workspace_id` из контекста.
3. **i18n.** Все тексты — через ключи, не хардкод. Язык берётся из контекста.
4. **Точка проверки доступа (gate).** Резерв заложить заранее: перед действием модуль может спросить «доступно ли это воркспейсу по тарифу/роли/состоянию». На пилоте gate пропускает всё, но место в контракте есть — чтобы платежи (Этап 9) не требовали переписывания.
5. **Ошибки.** Модуль не падает молча — возвращает понятный результат по стратегии ошибок (см. CONVENTIONS.md).
6. **Источник ввода прозрачен.** Модуль не должен зависеть от того, пришёл текст с клавиатуры или из расшифровки голоса.

## Открыто к детализации при сборке
Точные сигнатуры, формат payload и результата, как ядро выбирает модуль, как хранится состояние сессии между шагами.
