# Task Assistant Bot — Проектная концепция

> Статус: черновик v0.1 (этап понимания, не разработки)
> Стек: Telegram Bot API · Supabase (Postgres + pgvector + Edge Functions + Storage) · OpenAI (Whisper, embeddings, gpt-4o-mini)
> Тип: полностью новый, отдельный продукт для личного пользования.
> Ключевые принципы: модульность, human review-gate для базы знаний, файловая документация проекта.

---

## 1. Что мы строим

AI-бот в Telegram — личный помощник для ведения задач. Полностью новый, отдельный продукт для личного пользования.

Бот закрывает пять функциональных потребностей:

1. **Задачи и заметки** — хранение задач, заметок к ним, разбивка по темам и подрядчикам.
2. **Заметки на лету** — внесение заметок прямо во время встречи (в т.ч. голосом).
3. **Даты и календарь** — управление датами задач, синхронизация с Google Calendar (с заметками и вложениями).
4. **Голос** — распознавание речи для (а) диктовки заметок и (б) голосового управления задачами.
5. **База знаний по компании** — отдельный блок: данные по подрядчикам/компаниям, над которыми можно выполнять операции (пример: «составь прайс-лист от этой компании»). Работает на RAG, с обязательным review-gate.

### Кто пользователи
- **Сейчас:** один пользователь (стадия разработки и тестов).
- **В перспективе:** коммерческий мультитенантный продукт — продаётся разным, не связанным между собой людям/командам.
- **Вывод для архитектуры:** это мультитенантный SaaS. Изоляция — на уровне `workspace_id` в каждой строке **плюс** Row Level Security (RLS) в Postgres, чтобы данные одного клиента физически нельзя было увидеть из-под другого даже при ошибке в коде. Закладывается с первого дня. См. раздел 6.1.

## 2. Главный архитектурный принцип — нарезка на блоки

Цель: дорабатывать модули, не трогая основной код. Достигается за счёт того, что **ядро (core) ничего не знает о конкретных модулях**. Ядро умеет только:
- принимать апдейты от Telegram;
- определять намерение (команда / текст / голос / файл);
- держать состояние многошаговых диалогов (таблица `bot_sessions`);
- проверять права доступа.

Каждый модуль регистрируется в ядре через единый контракт и работает изолированно. Добавление модуля «Голос» или «Календарь» не требует правок в Tasks или Notes.

```
┌─────────────────────────────────────────────┐
│  CORE (роутер + сессии + права)               │
│  index.ts · router.ts · session.ts · auth.ts  │
└───────────────┬───────────────────────────────┘
                │ единый контракт модуля
   ┌────────────┼────────────┬──────────┬─────────────┐
   ▼            ▼            ▼          ▼             ▼
[Tasks]    [Notes]      [Voice]   [Calendar]   [Knowledge Base]
 задачи    заметки/    распозн.   синк дат +    база по компании
 темы/     встречи     + voice    файлы в       (отдельный блок,
 подрядч.  + файлы     команды    события       pgvector RAG)
```

### Контракт модуля (концептуально)
Каждый модуль реализует набор обработчиков:
- `commands` — какие команды он обслуживает (`/task`, `/note`, `/kb`...).
- `canHandle(intent)` — может ли обработать распознанное намерение.
- `handle(ctx)` — выполнить действие, имея доступ к контексту (пользователь, сессия, payload).

Ядро держит реестр модулей и направляет событие первому подходящему. Модуль не знает, пришёл ли текст с клавиатуры или из расшифровки голоса — это забота слоя Voice/Core.

## 3. Функциональные блоки

### 3.1 Core
Приём апдейтов, маршрутизация, состояние диалогов, права. Здесь же — middleware: идентификация пользователя, лимиты, логирование.

### 3.2 Tasks
- CRUD задач, статусы (открыта / в работе / выполнена / отложена).
- **Три независимые оси разбивки (всё структурными полями, без нейронки):**
  - `contractor_id` → **заказчик** (с кем: ООО «Барашек», оператор Иван). **Необязательное** — не у всякой задачи есть конкретный заказчик. Общая сущность с базой знаний.
  - `topic_id` → **тема/направление** (про что: Продакшен, Поставки, Полиграфия…). Справочник тем. **Есть всегда:** если тема не указана — падает в дефолтную тему **«Прочее»** (не NULL, а реальное значение справочника, чтобы фильтр «покажи прочее» работал штатно).
  - `category_id` → **вид работы** (что делаем: переговоры, продажи, маркетинг, поиск подрядчиков, сравнение цен…). **Одна на задачу.** Справочник категорий — основа аналитики.
  - **участники** (кто делает/участвует) → сущность `people` + связь many-to-many `task_participants` с ролью (исполнитель / участник / наблюдатель). **Дефолт:** исполнитель = сам пользователь (проставляется автоматически). Участник (внутренний человек) ≠ заказчик (внешний контрагент) — разные сущности. Закрывает запросы «все задачи, где участвовал X» точным фильтром, всегда полно.
- Темы и категории — разные оси: «переговоры ПРО продакшен» = `category = Переговоры` И `topic = Продакшен`. Не конкурируют.
- Фильтры — обычные запросы к БД, не нейронка: «по продакшену» = `topic_id = X`, «по Барашку» = `contractor_id = Y`, «сколько переговоров за месяц» = `category_id = Z`. Быстро, точно, дёшево.

### 3.3 Notes
- Заметки, привязанные к задаче.
- Режим «заметки во время встречи»: несколько заметок подряд в одну задачу/сессию.
- Заметка может быть текстовой или результатом расшифровки голоса (источник фиксируется).

### 3.4 Attachments (вложения)
- Файл / ссылка / скриншот / PDF / документ.
- Хранение файлов в Supabase Storage, метаданные — в таблице `attachments`.
- Полиморфная привязка: вложение может относиться к задаче или к заметке (см. риск в 6.4).

### 3.5 Voice
Два независимых режима:
- **Диктовка** (Phase 1, просто): голос → Whisper → текст → заметка.
- **Управление** (поздняя фаза, сложно): голос → текст → NLU (намерение + сущности) → подтверждение → действие. Обязателен шаг подтверждения перед изменением данных.

### 3.6 Calendar
- Phase 1: односторонний синк бот → Google Calendar. Задача с датой создаёт событие; заметки идут в описание, ссылки на вложения — туда же.
- Поздняя фаза: двусторонний синк (вебхуки, разрешение конфликтов, `google_event_id`).

### 3.7 Knowledge Base (отдельный блок)
- RAG над данными компании/подрядчиков: эмбеддинги (text-embedding-3-small), векторный поиск (pgvector), генерация ответа (gpt-4o-mini).
- Пример операции: «составь прайс-лист от компании X» → поиск записей с `contractor_id = X` → генерация.
- **Review-gate:** ничего не попадает в KB автоматически. Любой контент (из заметок, файлов, внешних источников) проходит ручное подтверждение пользователем перед добавлением.
- Архитектурно это **другой движок**, чем Tasks: Tasks — транзакционный CRUD, KB — семантический поиск. Не смешивать. Связь — только через `contractor_id`.

> **Принцип: три режима поиска — не всё нейронка.** Поиск по задачам/заметкам делится на три механизма, каждый для своего типа запроса:
> 1. **Структурный фильтр** (поля: заказчик, тема, категория, участники, даты, статусы) — обычный SQL. Для «задачи по Барашку / по продакшену / где участвовал Миша / за эту неделю». Быстро, точно, полно.
> 2. **Полнотекстовый поиск** (Postgres full-text + точное совпадение по строке/коду, ILIKE) — для конкретики: «где использовали РАЛ 9966», артикулы, номера, точные фразы. Векторный поиск тут НЕПРАВИЛЕН — он размывает точные коды и может вернуть «похожее» (РАЛ 9003) вместо точного.
> 3. **Векторный/семантический** (pgvector) — только для смысловых обобщений: «на чём застряли по продакшену», «что горит по Барашку и почему».
>
> Большинство реальных запросов закрывают режимы 1–2 (дёшево и точно). Нейронка (режим 3) — лишь для настоящих смысловых вопросов поверх отфильтрованного списка.

## 4. Сквозные сценарии (как блоки работают вместе)

1. **Голосовая заметка на встрече:** пользователь шлёт голосовое → Voice расшифровывает → Notes создаёт заметку в активной задаче → (опц.) вложения подцепляются к той же задаче.
2. **Задача с дедлайном → календарь:** Tasks создаёт задачу с датой → Calendar создаёт событие с заметками и ссылками на файлы.
3. **Прайс от подрядчика:** KB получает запрос → RAG-поиск по `contractor_id` → черновик прайса → пользователь правит. (Если результат сохраняется обратно в KB — проходит review-gate.)
4. **Голосовое управление:** «закрой задачу про макет» → Voice → NLU находит кандидатов → бот показывает совпадение и просит подтвердить → только потом меняет статус.

## 4-бис. Расширенные возможности: что в фундамент сейчас, что модулем потом

Главный принцип: **в фундамент закладываем то, без чего потом придётся переписывать схему или терять исторические данные. Остальное — модули поверх готового фундамента.**

### A. Заложить в фундамент СЕЙЧАС (иначе переделка с нуля)

| Возможность | Что это значит в схеме | Почему сейчас |
|---|---|---|
| Темы/направления (Продакшен, Поставки) с вкл/выкл | Справочник `topics` + флаг видимости; задача через `topic_id`; дефолт «Прочее» | Иначе переделка фильтров; тема ≠ заказчик ≠ категория |
| Участники/исполнители задач | Сущность `people` + связь `task_participants` с ролью; дефолт-исполнитель = пользователь | Ось «кто» нужна для полного поиска по людям; участник ≠ заказчик |
| Полнотекстовый поиск по задачам/заметкам | FTS-индексы Postgres + точное совпадение по строке | Поиск конкретики (коды, артикулы) — отдельно от векторного |
| Категории-типы для аналитики | Справочник `categories`; `category_id` на задаче (одна на задачу) | Поле-классификатор нужно с первого дня, иначе аналитика неполная |
| Учёт времени | Таблица `time_entries` (старт/стоп или ручной ввод) | Если не собирать данные сразу — ретроспективы за прошлый период не будет |
| Подзадачи | `parent_task_id` (self-reference) у задачи | Одно поле сейчас → модуль подзадач позже без миграции |
| Профили подрядчиков | Таблица `contractor_contacts` (тип + значение): соцсети, почты, телефоны, мессенджеры | Подрядчик — центральная сущность; контактов много и разных |
| Пинги/напоминания ко времени | Таблица `reminders` + планировщик (pg_cron / Edge Function по расписанию) | Инфраструктурное решение, предусмотреть в архитектуре сразу |
| Историчность / админка / ретроспектива | Soft-delete (`archived_at`/`deleted_at`), журнал изменений статусов | Если удалять физически — ретроспективу не построить |
| Услуги/прайс-позиции | Сущность `services` / `price_items` у подрядчика (с ценой) | Прайс — заявленная цель; на ней же завязан финансовый модуль |

### B. Можно отложить (модуль поверх фундамента, без переделки)

| Возможность | Тип | Зависит от фундамента |
|---|---|---|
| Экран «сегодня + просроченные» при входе | Запрос по `due_date` | Только наличие даты у задачи (уже есть) |
| Нейронка «спросить по задачам в чате» | RAG-модуль (тот же движок, что и KB) | Хранение текстов (уже есть) |
| Отправка писем из бота | Интеграция (почтовый API + OAuth) | Минимально |
| Прайс-лист по чек-листу → PDF по шаблону | Модуль-процесс: исходники услуг → чек-лист → итог → PDF | Сущность услуг (закладывается в A) |
| Лист закупок / вишлист | Лёгкий модуль-список | Не трогает фундамент |
| Калькулятор бюджета и поступлений | Финансовый модуль — **отдельный брейншторм** | Денежные поля у услуг/подрядчика (закладываются в A) |

### Заметки по отдельным пунктам
- **Пинги ко времени — да, бот сможет.** Механизм: планировщик (Supabase `pg_cron` или Edge Function по cron) проверяет `reminders` и шлёт сообщения в Telegram. Закладываем таблицу и расписание в архитектуру, включаем когда удобно.
- **Нейронка по задачам и база знаний — один движок.** Оба используют эмбеддинги + pgvector + генерацию. Разница только в источнике данных (задачи/заметки против KB-записей). Это экономит работу.
- **Прайс-лист — это рабочий процесс, не просто экспорт.** Бот ведёт по чек-листу (чтобы ничего не забыть), собирает итог, генерит PDF по шаблону. Опирается на сущность услуг подрядчика.
- **Калькулятор бюджета — выносим в отдельный брейншторм.** В схеме лишь закладываем, что у услуг/подрядчика есть денежные поля.

Дефолты выбраны на минимизацию риска: сначала ценность без риска порчи данных, сложное — потом. Меняется без переписывания структуры.

### Phase 0 — Фундамент
- Supabase-проект, схема данных (сразу мультитенантная: `workspace_id` + RLS), `bot_sessions`.
- Core: роутер, сессии, идентификация пользователя.
- Telegram-бот, базовый эхо/команды.
- **Заложенные в схему (без UI) на будущее, чтобы не переделывать:** темы с флагом видимости (вкл/выкл) и дефолтом «Прочее», категории (справочник + `category_id`), участники (`people` + `task_participants`), `time_entries`, `parent_task_id` (подзадачи), контакты подрядчика, `reminders`, soft-delete + журнал статусов, сущность услуг/прайс-позиций, FTS-индексы для полнотекстового поиска. Подробности — раздел 4-бис.

### Phase 1 — Ядро ценности
- Tasks: CRUD, заказчики, темы (с вкл/выкл), категории, статусы, списки/фильтры.
- Экран входа: «сегодня + просроченные».
- Notes: заметки к задачам, режим «во время встречи».
- Attachments: файлы/ссылки/PDF/скрины в Storage.
- Voice — **только диктовка**.
- Calendar — **односторонний** синк бот → Google Calendar.
- Reminders: пинги по задачам ко времени (планировщик).

### Phase 2 — База знаний и интеллект
- KB-движок: эмбеддинги, pgvector-поиск, генерация.
- Review-gate.
- Нейронка «спросить по задачам» (тот же движок).
- Операции над данными подрядчика; профили подрядчиков с контактами.
- Прайс-лист: чек-лист → итог → PDF по шаблону.
- Аналитика по категориям; учёт времени (UI поверх собранных данных).

### Phase 3 — Сложное и рискованное
- Голосовое **управление** задачами (NLU + подтверждение).
- **Двусторонний** календарь (вебхуки, конфликты).
- Подзадачи (модуль поверх заложенного `parent_task_id`).
- Отправка писем из бота (почтовый API + OAuth).
- Мультиюзер на уровне UI (приглашения, роли, биллинг).

### Отдельный трек (вне фаз, требует своего брейншторма)
- Калькулятор бюджета и поступлений от подрядчиков.
- Лист закупок / вишлист.

## 6. Узкие места и риски (отсортированы по стоимости исправления потом)

### 6.1 Мультитенантность с первого дня в схеме — ДОРОГО исправлять потом
Продукт продаётся не связанным между собой клиентам. Если таблицы создать без `workspace_id`, переход на мультитенантность = переписывание половины запросов + реальный риск утечки данных одного клиента другому. **Решение:** каждая строка (задача, заметка, вложение, KB-запись, подрядчик) с самого начала несёт `workspace_id`, и поверх включён **Row Level Security**: политики БД отсекают доступ к чужим воркспейсам на уровне Postgres, независимо от логики приложения. Сейчас всегда один воркспейс — накладных расходов почти ноль.

### 6.2 Голосовое управление без подтверждения — ОПАСНО (порча данных)
Распознавание ошибается. «Закрой задачу X» может закрыть не ту. **Решение:** обязательный шаг подтверждения перед любым изменением, инициированным голосом. Вынесено в Phase 3.

### 6.3 Двусторонний календарь недооценён по сложности — ДОРОГО по времени
Вебхуки, разрешение конфликтов, обработка «событие удалили/изменили в Google». На порядок сложнее одностороннего. **Решение:** Phase 1 — только односторонний.

### 6.4 Google OAuth — требует отдельного времени
Интеграция с Google Calendar означает прохождение OAuth-флоу: настройка проекта в Google Cloud, согласие пользователя, хранение и обновление токенов. Это не «быстрая задача» — заложить отдельное время на настройку и тестирование.

### 6.5 Полиморфные вложения — целостность на уровне приложения
В Postgres нет внешнего ключа на полиморфную пару `(entity_type, entity_id)`. **Решение:** либо приложение следит за целостностью (проще, гибче), либо отдельные таблицы связей под каждый тип (строже, больше кода). Решить на этапе схемы.

### 6.6 Парсинг дат из естественной речи — СРЕДНЕ, влияет на UX
«Перенеси на следующий вторник», «через две недели» — нужен надёжный парсер дат с учётом таймзоны пользователя. Влияет и на текст, и на голос.

### 6.7 Стоимость OpenAI при росте — ОТЛОЖЕННЫЙ риск
