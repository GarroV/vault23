**Критерий завершения этапа:** есть пакет контекста, можно писать первый промт для Gemini.

> **Два сквозных принципа, которые ОБЯЗАТЕЛЬНО фиксируются в пакете контекста (иначе их почти невозможно добавить потом):**
>
> **(А) Мультиязычность с первой строки (i18n) — РУ + АНГ сразу.** Бот разрабатывается сразу двуязычным: русский и английский.
> - Все интерфейсные тексты бота — через ключи (`task_created`), не хардкодом, в отдельных языковых файлах `ru` и `en`. Оба заполняются по мере разработки каждого блока (не «потом»).
> - **Автоопределение языка:** из `language_code` Telegram. Правило — `ru` → русский, всё остальное (и отсутствие кода) → **английский (дефолт/фолбэк)**.
> - **Ручная смена + запоминание:** автоопределение — лишь дефолт при первом запуске. Пользователь может переключить язык, выбор сохраняется в профиле (поле «язык» у пользователя/воркспейса).
> - **Переводим только интерфейс, НЕ данные пользователя.** Названия задач, заметки, темы остаются на языке ввода — их не трогаем. i18n касается только текстов самого бота.
> - Архитектура остаётся расширяемой: добавить третий язык = добавить файл, не трогая код.
> Правило → `CONVENTIONS.md`.
>
> **(Б) Логика отдельно от бота.** Бизнес-логика живёт в общем слое (внутреннее API на Edge Functions + база), Telegram-бот — лишь один из клиентов, который этот слой дёргает. Бот НЕ содержит логику, только принимает ввод и зовёт общий слой. Тогда сайт/приложение/другие формы завтра — это ещё один клиент к тому же слою, а не переписывание. Правило → `MODULE_CONTRACT.md`.

## ⚠️ ОТДЕЛЬНАЯ ПРОРАБОТКА ДО ЭТАПА 9: Платежи и доступы

Перед стартом Этапа 9 создаётся документ `BILLING_AND_ACCESS_DESIGN.md`. Это нельзя проектировать на ходу. Минимум, что в нём закрывается:

| Вопрос | Почему критично |
|---|---|
| Платёжный провайдер и страны/валюты | Определяет всю интеграцию; нельзя поменять задним числом легко |
| Матрица «функция × тариф» | Каждое действие должно знать, доступно ли оно — нужна единая точка проверки |
| Модель ролей внутри воркспейса | Доступы — отдельный слой поверх RLS-изоляции |
| Единая точка проверки доступа (gate) | Без неё проверки расползутся по всему коду |
| Обработка сбоев оплаты | Деньги требуют надёжности выше обычной (двойное списание, недоставка вебхука) |
| Что с данными при понижении/отмене тарифа | Юридически и по UX чувствительно |
| Неразглашение платёжных данных (делегировать провайдеру) | Соответствие требованиям, чтобы не хранить карты у себя |
| Лимиты как часть тарифов (строка в матрице функция×тариф) | У каждого плана свои лимиты на распознавание/вопросы/эмбеддинги; отдельно от защитного потолка |
| Антифрод | Двухсоставный: платёжный (краденые карты, chargeback) — закрывает провайдер; ресурсный (повторные триалы, накрутка лимитов, использование OpenAI-ключа как бесплатного API) — закрываешь ты лимитами и состояниями |

### Антифрод — два разных врага

| Тип | Кто закрывает | Чем |
|---|---|---|
| **Платёжный** (краденые карты, оспаривание платежей) | Провайдер (Stripe/Paddle) | Их встроенный антифрод — ещё аргумент не катать платежи самому |
| **Ресурсный** (повторные триалы, накрутка бесплатных лимитов, использование твоего OpenAI как бесплатного API) | Ты сам | Лимиты, состояния доступа, отслеживание подозрительных паттернов, защита триала от повторов |

### Лимиты токенов — две роли одного механизма

Один механизм «лимит токенов» служит двум целям, но управляется раздельно:
- **Защитный потолок** — оборона от перерасхода, поднимаешь вручную конкретному клиенту.
- **Тарифный лимит** — часть продаваемого плана, меняется сменой тарифа, живёт строкой в матрице «функция × тариф».

Не сваливать в одну настройку, иначе непонятно, где «поблажка другу», а где «оплаченный лимит».

### Состояния доступа воркспейса (не «вкл/выкл», а жизненный цикл)

Доступ — это не флаг «платит/не платит», а несколько состояний, между которыми воркспейс перемещается. Gate проверяет именно состояние:

| Состояние | Что значит | Доступ |
|---|---|---|
| **Триал** | Идёт пробный период, отсчёт времени | Полный (или по правилам триала) |
| **Активна** | Оплачено, подписка действует | Полный по тарифу |
| **Просрочка (grace)** | Оплата не прошла, но даём буфер несколько дней | Ещё работает — шанс обновить карту |
| **Приостановлен** | Отключён за неуплату, данные сохранены | Заблокирован, данные целы |
| **Отменён** | Клиент ушёл сам | Заблокирован |

**Принцип:** между «не заплатил» и «отключить» всегда есть буфер (grace-период). Рубить доступ моментально нельзя — карта могла просто протухнуть, потеряешь клиента зря.

### Пробный период (триал) — отдельная механика

- **С картой или без?** Без карты — ниже барьер входа, больше нецелевых. С картой вперёд — выше конверсия, авто-списание по окончании. Решить до кода.
- **Что в конце триала?** Полная блокировка / урезанный бесплатный план / заморозка с сохранением данных. Влияет на возврат клиента.
- **Защита от повторных триалов** — чтобы один человек не заводил новый воркспейс каждые N месяцев ради вечной халявы. Продумать отслеживание.

### Админка доступов — ДВА разных интерфейса (не путать)

| Админка | Кто | Что может |
|---|---|---|
| **Админка владельца продукта** (твоя) | Ты | Видеть всех клиентов, вручную продлить триал, дать бесплатный доступ, отключить за неуплату, смотреть статусы оплат, разруливать споры. Твой пульт бизнеса. |
| **Личный кабинет клиента** | Покупатель | Управлять своей подпиской: тариф, смена плана, обновление карты, приглашение сотрудников и раздача им ролей. Видит только себя. |

**Не сливать в один интерфейс** — разные права, разные пользователи. Легко случайно объединить на проектировании, больно разделять потом.

**Связь с заложенным фундаментом:** `workspace_id` + RLS = доступ привязан к воркспейсу (сменил статус — включились/выключились все его данные и функции разом). Gate в ядре смотрит на состояние воркспейса и пускает/не пускает. Сегодняшние требования — это наполнение уже зарезервированного места логикой состояний, а не новая архитектура.
