# ЧАСТЬ I. КОНЦЕПЦИЯ

## Что строим
AI-бот в Telegram — личный помощник для ведения задач. Полностью новый, отдельный продукт для личного пользования, в перспективе — коммерческий мультитенантный SaaS на продажу разным, не связанным между собой людям/командам.

## Стек (зафиксирован)
- Supabase: Postgres + pgvector + Edge Functions + Storage
- TypeScript / Deno (рантайм Edge Functions)
- Telegram Bot API
- OpenAI: Whisper (распознавание речи), text-embedding-3-small (эмбеддинги), gpt-4o-mini (генерация)

Решение по стеку: дёшево для старта и адекватно для роста — не «времянка», а фундамент, дорастающий до коммерческого продукта (free-план → платный план Supabase). Подтверждено как осознанный выбор против альтернатив (Python/VPS/Firebase).

## Ключевые принципы
1. **Модульность:** ядро не знает о конкретных модулях; модули регистрируются через единый контракт. Дорабатывать модуль, не трогая остальное.
2. **Review-gate:** ничего не попадает в базу знаний автоматически — только через ручное подтверждение пользователя.
3. **Мультитенантность с первого дня:** `workspace_id` в каждой строке + Row Level Security (RLS).
4. **Логика отдельно от бота:** бизнес-логика в общем слое; бот/сайт/приложение — сменные клиенты.
5. **i18n с первой строки:** тексты через ключи, бот сразу двуязычный РУ+АНГ.
6. **Файловая документация** для непрерывности между сессиями.

## Кто пользователи
- Сейчас: один пользователь (пилот, тесты).
- Перспектива: разные несвязанные клиенты (мультитенантный SaaS).

# ЧАСТЬ II. ТЕХНИЧЕСКОЕ ЗАДАНИЕ

## Модуль ядра (Core)
Приём апдейтов Telegram, маршрутизация (намерение: команда/текст/голос/файл), состояние многошаговых диалогов (`bot_sessions`), идентификация пользователя + воркспейс, права/доступы (gate). Реестр модулей по контракту.

## Модуль Tasks — четыре оси разбивки
Все оси — структурные поля, фильтрация БЕЗ нейронки.
- `contractor_id` → **заказчик** (с кем). Необязательное.
- `topic_id` → **тема** (про что: Продакшен, Поставки…). Есть всегда; дефолт «Прочее» (реальное значение справочника, не NULL).
- `category_id` → **вид работы** (что делаем: переговоры, продажи…). Одна на задачу. Справочник — основа аналитики.
- **участники** → `people` + `task_participants` (many-to-many, роль: исполнитель/участник/наблюдатель). Дефолт-исполнитель = пользователь. Участник (внутренний) ≠ заказчик (внешний).

Статусы задач: открыта / в работе / выполнена / отложена.

## Три режима поиска (не всё — нейронка)
1. **Структурный фильтр** (SQL по полям) — «по Барашку», «по продакшену», «у Миши», «за неделю». Быстро, точно, полно.
2. **Полнотекстовый** (Postgres FTS + точное совпадение) — конкретика, коды, артикулы («РАЛ 9966»). Векторный тут НЕПРАВИЛЕН.
3. **Векторный** (pgvector) — только смысловые обобщения («на чём застряли по продакшену»).

## Модуль Notes
Заметки к задачам, режим «во время встречи» (серия заметок в сессию), источник фиксируется (текст/голос).

## Модуль Attachments
Файл/ссылка/скрин/PDF/документ. Хранение в Supabase Storage, метаданные в `attachments`. Полиморфная привязка (задача/заметка) — целостность на уровне приложения.

## Модуль Voice
- Диктовка (Phase 1): голос → Whisper → текст → заметка.
- Управление (поздняя фаза): голос → текст → NLU → **обязательное подтверждение** → действие.

## Модуль Calendar
- Phase 1: односторонний синк бот → Google Calendar (с заметками/файлами).
- Поздняя фаза: двусторонний (вебхуки, конфликты).

## Модуль Knowledge Base (база знаний)
RAG над данными подрядчиков: эмбеддинги + pgvector + генерация. Пример: «прайс от компании X» = поиск по `contractor_id` → генерация. Review-gate обязателен. Другой движок, чем Tasks; связь — через `contractor_id`.

## Прочие сущности
Профили подрядчиков + `contractor_contacts` (соцсети/почты/телефоны/мессенджеры). Услуги/прайс-позиции. `time_entries` (учёт времени). `reminders` (напоминания + планировщик pg_cron/Edge Function). Подзадачи через `parent_task_id`.

## Сквозные требования (заложить в фундамент)
- **Мультитенантность:** `workspace_id` + RLS на каждой таблице.
- **i18n:** все тексты через ключи, файлы `ru`/`en` заполняются сразу; автоопределение по `language_code` (ru→рус, иначе→англ-дефолт); ручная смена + сохранение в профиль; данные пользователя НЕ переводятся.
- **Идентичность:** не делать Telegram ID первичным ключом; внутренний ID пользователя + таблица способов входа (Telegram/почта/Google). Связывание — через подтверждение владения (код на почту), всегда из уже подтверждённой личности.
- **Время:** хранить в UTC; поле TZ в профиле заморожено (пояса как фичу пока не поддерживаем — Telegram пояс не отдаёт, спрашивать = плохой UX).
- **Удаление:** два уровня — мягкое для задач (корзина), полное и бесповоротное для аккаунта (право на забвение). Остаётся только обезличенная аналитика (необратимый разрыв связи с личностью).
- **Согласие:** механизм за feature flag (выключен на пилоте), поле факта согласия в профиле сразу.
- **Идемпотентность:** запоминать обработанные `update_id` (защита от дублей).
- **Стратегия ошибок:** fail gracefully, 3 класса (пользователя/временная системная+retry/баг), не терять ввод, логи без чувствительного.
- **Лимиты OpenAI на воркспейс:** учёт расхода токенов с первого дня; защитный потолок отдельно от тарифного лимита.

# ЧАСТЬ III. МОДЕЛЬ ДАННЫХ (предварительный список таблиц)

Полная `DATA_MODEL.md` собирается на шаге 1.1. Текущий состав:

- `workspaces` — тенант; состояние доступа (триал/активна/просрочка/приостановлен/отменён), тариф
- `users` — внутренний ID, язык, поле согласия, TZ (заморожено)
- `auth_methods` — способы входа (Telegram ID / почта / Google), привязаны к user
- `contractors` — заказчики/подрядчики (общая сущность Tasks ↔ KB)
- `contractor_contacts` — контакты подрядчика (тип + значение)
- `topics` — темы (дефолт «Прочее», флаг видимости)
- `categories` — виды работ
- `tasks` — задачи (4 оси, статусы, даты, `parent_task_id`, soft-delete)
- `people` — участники/члены команды
- `task_participants` — связь задача↔участник + роль
- `notes` — заметки (привязка к задаче, источник)
- `attachments` — вложения (полиморфно, Storage)
- `time_entries` — учёт времени
- `reminders` — напоминания
- `services` — услуги/прайс-позиции (денежные поля под будущий калькулятор)
- `kb_entries` — база знаний (pgvector)
- `bot_sessions` — состояние диалогов
- (+ таблица учёта расхода токенов OpenAI)

Все содержательные таблицы несут `workspace_id` + RLS. Время — UTC. Удаляемые сущности — soft-delete.

# ЧАСТЬ IV. ПЛАН РАЗРАБОТКИ

## Роли в разработке
- 🔴 **Claude** — архитектура, безопасность, проверка, стыковка. НЕ отдаётся Gemini: RLS, ядро, маршрутизация, review-gate, контракт, конфликты.
- 🟢 **Gemini** (корпоративный, без лимитов) — объёмные/шаблонные блоки по готовой спеке: CRUD, справочники, миграции, обёртки над API, форматирование.
- 🟡 **Совместно** — Gemini черновик, Claude дорабатывает логику.
- Принцип отдачи Gemini: **объём и шаблонность — да, хитрая логика и безопасность — нет.**
- Конвейер: пакет контекста → меткий промт → генерация → адаптация и проверка Claude → интеграция.

## Этапы (последовательно)

**Этап 0 — Окружение:** репозиторий, Supabase-проект, CLI, бот в Telegram, ключи в секреты, файлы документации (CLAUDE.md и др.).

**Этап 1 — Пакет контекста (до любой генерации):** `DATA_MODEL.md`, `MODULE_CONTRACT.md`, `CONVENTIONS.md`, промт-шаблон для Gemini. Здесь же фиксируются сквозные принципы (i18n, логика не в боте).

**Этап 2 — Схема БД и безопасность:** миграции таблиц (🟢), применение (🔴), дефолты («Прочее», категории), **RLS-политики (🔴)**, **тест изоляции (🔴, блокер)**.

**Этап 3 — Ядро:** вебхук+эхо (🟢), идентификация+воркспейс (🔴), язык+i18n-каркас (🟡/🔴), реестр модулей (🔴), роутер (🔴), сессии (🟡).

**Этап 4 — Tasks:** справочники (🟢), people+participants (🟢), CRUD задач (🟢), команды/кнопки (🟢), фильтры по осям (🟡), «сегодня+просроченные» (🟢), интеграционная проверка (🔴).

**Этап 5 — Заметки, файлы, голос:** Notes CRUD (🟢), режим встречи (🟡), Attachments (🟡), полиморфизм (🔴), Voice диктовка (🟢), голос→заметка (🟡).

**Этап 6 — Напоминания, лимиты, устойчивость:** reminders (🟢), планировщик (🟡), **лимиты OpenAI + учёт токенов (🔴)**, обработка недоступности сервисов (🔴).

**Этап 7 — База знаний:** профили подрядчиков (🟢), услуги (🟢), эмбеддинги (🟢), FTS (🟢), векторный поиск (🟡), **review-gate (🔴)**, нейронка по задачам (🟡), прайс→PDF (🟡), аналитика+время (🟢).

**Этап 8 — Календарь и сложное:** Google OAuth (🔴), односторонний синк (🔴), голосовое управление+подтверждение (🔴), двусторонний календарь (🔴), подзадачи (🟡), отправка писем (🟡).

**Этап 9 — Коммерциализация:**
- 9A Платежи и доступы (проектировать ДО кода через `BILLING_AND_ACCESS_DESIGN.md`): провайдер, матрица функция×тариф, роли в воркспейсе, единая точка gate, интеграция+сбои оплаты, карты не хранить.
- 9B Обвязка: лендинг, личный кабинет, тарифы, справка/онбординг, email-уведомления, админка токенов, админка языков.

## Состояния доступа воркспейса
Триал → Активна → Просрочка (grace) → Приостановлен → Отменён. Gate проверяет состояние. Между «не заплатил» и «отключить» — буфер. Триал: с картой/без, что в конце, защита от повторов. Админка доступов двухуровневая: владельца продукта (все клиенты) и клиента (своя подписка).

# ЧАСТЬ VII. РЕШЁННЫЕ ВОПРОСЫ (зафиксированные решения)

1. **Стек** — Supabase + TS/Deno + Telegram + OpenAI. Дёшево для старта, дорастает до продукта.
2. **Изоляция** — `workspace_id` + RLS (мультитенантность с первого дня).
3. **Разбивка задач** — 3 структурные оси + участники (заказчик/тема/вид/участники). Тема дефолт «Прочее». Категория одна на задачу.
4. **Поиск** — три режима (структурный/полнотекстовый/векторный), фильтрация не нейронкой.
5. **Участники** — отдельная ось, `people`+`task_participants`, дефолт-исполнитель = пользователь.
6. **i18n** — РУ+АНГ сразу, автоопределение (не рус→англ), ручная смена, данные не переводятся.
7. **Идентичность** — внутренний ID + способы входа; связывание через подтверждение владения.
8. **Часовые пояса** — UTC внутри, фича поясов заморожена (Telegram не отдаёт, спрашивать = плохой UX).
9. **Удаление** — мягкое для задач, полное для аккаунта; остаётся обезличенная аналитика.
10. **Согласие** — механизм за feature flag (выкл на пилоте), поле в схеме сразу.
11. **Файлы** — лимит 20 МБ, сообщение об отказе; платное хранилище на будущее.
12. **Идемпотентность** — по `update_id`.
13. **Ошибки** — fail gracefully, 3 класса, не терять ввод.
14. **Почта** — НЕ в онбординге; тариф на воркспейсе, не на почте; появится при оплате/для сайта.
15. **Онбординг** — минимальный (приветствие + примеры), остальное по моментам.
16. **Платежи** — отдельная проработка ДО кода (`BILLING_AND_ACCESS_DESIGN.md`), карты не хранить (провайдер).
17. **Gemini** — объёмная рутина да, безопасность/ядро нет.
18. **Бэкапы** — план сейчас, имплементация после пилота; проверить восстановление хотя бы раз.
