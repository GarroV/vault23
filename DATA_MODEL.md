# DATA_MODEL.md — модель данных

> Источник правды по схеме БД. Заполняется на шаге 1.1 (🔴 Claude).
> Правила: каждая содержательная таблица несёт `workspace_id` + RLS. Время в UTC. Удаляемые сущности — soft-delete. Telegram ID не первичный ключ пользователя.
> Статус: ЗАГОТОВКА — структура намечена, поля детализируются при сборке.

## Конвенции схемы
- Первичные ключи — внутренние UUID, не внешние ID (Telegram и т.п.).
- `workspace_id` на каждой содержательной таблице, FK на `workspaces`.
- Временные поля — `timestamptz` в UTC.
- Soft-delete — поле `archived_at` / `deleted_at` (где применимо).
- Имена таблиц — множественное число, snake_case.

## Таблицы (план)

### workspaces
Тенант. Поля (намечено): id, название, состояние доступа (триал/активна/просрочка/приостановлен/отменён), тариф, даты, лимиты.

### users
Пользователь как личность. Поля: внутренний id, язык (ru/en), поле факта согласия (что/когда/версия), tz (заморожено), даты.

### auth_methods
Способы входа, привязанные к user. Поля: id, user_id, тип (telegram/email/google), значение, подтверждён ли, дата.

### contractors
Заказчики/подрядчики (общая сущность Tasks ↔ KB). workspace_id, название, поля профиля.

### contractor_contacts
Контакты подрядчика. contractor_id, тип (соцсеть/почта/телефон/мессенджер), значение.

### topics
Темы. workspace_id, название, флаг видимости. Дефолт «Прочее» (сидируется).

### categories
Виды работ (для аналитики). workspace_id, название.

### tasks
Задачи. workspace_id, заголовок, описание, contractor_id (nullable), topic_id (дефолт «Прочее»), category_id, parent_task_id (nullable, подзадачи), статус, due-дата (UTC), soft-delete, даты.

### people
Участники/члены команды. workspace_id, имя, связь с user (если внутренний).

### task_participants
Связь задача↔участник many-to-many. task_id, person_id, роль (исполнитель/участник/наблюдатель).

### notes
Заметки. workspace_id, task_id, текст, источник (текст/голос), даты.

### attachments
Вложения (полиморфно). workspace_id, entity_type (task/note), entity_id, тип файла, ссылка в Storage, метаданные. Целостность — на уровне приложения.

### time_entries
Учёт времени. workspace_id, task_id, начало/конец или длительность.

### reminders
Напоминания. workspace_id, на что ссылается, время (UTC), статус.

### services
Услуги/прайс-позиции. workspace_id, contractor_id, название, денежные поля (под будущий калькулятор бюджета).

### kb_entries
База знаний (RAG). workspace_id, contractor_id, контент, эмбеддинг (pgvector), статус подтверждения (review-gate).

### bot_sessions
Состояние многошаговых диалогов. workspace_id, user_id, состояние, данные шага.

### token_usage
Учёт расхода OpenAI. workspace_id, тип операции, объём токенов, дата. (Для лимитов и единичной экономики.)

## RLS
Для каждой таблицы выше — политика изоляции по `workspace_id`. Пишется вручную (🔴), не Gemini. Тест изоляции (два воркспейса) — блокер перед движением дальше.

## Открыто к детализации при сборке
Точные типы полей, индексы (включая FTS-индексы и pgvector), внешние ключи и каскады, сидирование справочников.
