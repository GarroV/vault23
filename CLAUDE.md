# CLAUDE.md — правила проекта Task Assistant Bot

> Этот файл Claude Code читает автоматически в начале каждой сессии.
> Здесь только короткие всегда-действующие правила. Полный контекст — в `MASTER_PROJECT.md` (прочитать по запросу).

## Что это за проект
AI-бот в Telegram для ведения задач. Мультитенантный SaaS на продажу разным несвязанным клиентам.
Стек: Supabase (Postgres + pgvector + Edge Functions + Storage) · TypeScript/Deno · Telegram Bot API · OpenAI.

## Архитектурные правила (нерушимые)

1. **Мультитенантность.** Каждая содержательная таблица несёт `workspace_id`. На каждой — RLS-политика изоляции. Никогда не полагаться на фильтрацию только в коде.

2. **Логика не в боте.** Бизнес-логика живёт в общем слое (Edge Functions + БД). Telegram-бот — лишь один из клиентов, который дёргает этот слой. Бот принимает ввод и зовёт логику, но не содержит её. Цель: завтра сайт/приложение — ещё один клиент, не переписывание.

3. **Модульность.** Ядро не знает о конкретных модулях; модули регистрируются через контракт (`MODULE_CONTRACT.md`). Дорабатывать модуль, не трогая остальные.

4. **Review-gate.** Ничего не попадает в базу знаний автоматически — только через ручное подтверждение пользователя.

5. **Идентичность.** Telegram ID — НЕ первичный ключ пользователя. Внутренний ID + таблица способов входа (Telegram/почта/Google). Связывание способов входа — через подтверждение владения (код на почту), всегда из уже подтверждённой личности.

## Правила работы с данными

6. **Время — в UTC.** Всегда хранить в UTC. Поле TZ в профиле есть, но заморожено (пояса как фичу пока не поддерживаем). Отображение фиксированное.

7. **Удаление двухуровневое.** Задачи — мягко (soft-delete, корзина). Аккаунт — полно и бесповоротно (право на забвение). Остаётся только обезличенная аналитика с необратимым разрывом связи с личностью.

8. **Миграции.** Изменение структуры БД на живых данных — только через версионированную миграцию (Supabase CLI), не руками в прод. У каждой: имя, откат, дефолты для существующих строк, проверка на копии. Не удалять колонки/таблицы с данными без явного бэкапа.

## Правила кода

9. **i18n с первой строки.** Все интерфейсные тексты — через ключи, не хардкодом. Файлы `ru` и `en`, оба заполняются сразу при разработке блока. Автоопределение: `language_code` ru→русский, иначе→английский (дефолт/фолбэк). Данные пользователя (задачи, заметки) НЕ переводятся.

10. **Стратегия ошибок.** Бот никогда не молчит и не падает. Пользователю — только понятные сообщения, техника — в лог. Три класса: ошибка пользователя (объяснить, что исправить) / временная системная (сообщить + авто-retry пару раз) / баг (извиниться + детали в лог). Не терять ввод: сначала сохранить сырое, потом обрабатывать. В логах — без чувствительного содержимого заметок.

11. **Идемпотентность.** Запоминать обработанные `update_id` Telegram, игнорировать повтор (защита от дублей).

12. **Фильтрация — не нейронкой.** Выборка задач — структурными полями (SQL) или полнотекстовым поиском. Векторный поиск (pgvector) — только для смысловых обобщений, не для фильтров и не для точных кодов/артикулов.

## Правила разделения труда с Gemini

13. **Gemini — объёмная рутина по готовой спеке** (CRUD, справочники, миграции, обёртки над API, форматирование). Каждый блок проходит проверку и стыковку.

14. **НЕ отдавать Gemini:** RLS-политики, ядро/маршрутизацию, контракт модуля, review-gate, логику конфликтов, всё про безопасность и деньги. Это пишем и проверяем вручную.
